Positive closure now advances a node's source fingerprint and log baseline only when every enforced rule is freshly verified against the current code in this run and the node was not gated. A stored approval from an earlier run can no longer stand in for re-verification, so a source change can never be recorded as reviewed without an actual review. ## [2026-06-13T07:02:06.114Z]
When a component lists a file that cannot be read, filling and closing verification state must fail closed rather than crash or quietly advance. Computing a component's source fingerprint over an unreadable file now raises a typed error; the fill and positive-closure steps catch it and decline to advance that component's recorded fingerprint or log baseline, leaving the component blocked rather than aborting the whole command with an unclassified error or closing over content that was never verified. The reasoning is that an unreadable listed file is a hard error for the author to fix, never grounds to record a passing state.
## [2026-06-13T07:27:16.966Z]
Agent-facing diagnostic messages (the what/why/next structure that tells an agent what went wrong and how to fix it) are formatted by the command-line layer, not by the engine. The fill stage previously built and wrote these messages itself; that formatting work has been moved out. The engine now emits each diagnostic as a structured object carrying the three fields and hands it to a sink supplied by the caller, which renders it; plain progress lines (counts, per-pair status) stay as direct writes because they are status, not remediation guidance. The reason for the split is that keeping all message formatting in one layer means presentation can change in a single place, and the engine stays a pure producer of structured results that any front end can render.

## [2026-06-14T22:16:34.150Z]
The fill stage no longer runs or persists the architecture relation-conformance check. That check became a live computation performed on every read of the graph, so running it again during the verdict-filling stage and writing its outcome into the committed lock would duplicate work and store a result that is always recomputed. The garbage-collection step that pruned stale relation results from the lock is likewise removed, because there is no longer any relation result stored to prune. Verdict filling now concerns itself only with the rule verdicts it actually persists.
## [2026-06-16T09:52:38.910Z]
Two changes. LLM verdicts are now written to the lock the moment each pair completes, the same way deterministic checks already behaved, so interrupting a long approve keeps every finished verdict and the next run resumes only the rest; previously they were buffered and flushed only after the whole tier batch, so an interrupt discarded all completed reviewer work. Separately, the mandatory-log gate is now all-or-nothing: if any node needs a fresh log entry, approve stops and approves nothing rather than partially filling. The per-provider secrets merge was also removed because the yg-secrets overlay is now applied once at config parse time.

Run the companion resolver once per unit before the reviewer is called, inject the resolved paired files into that unit's prompt, and fail closed on any resolution problem so a torn or unresolvable pairing never costs a review and never records a verdict over an incomplete prompt. Resolving before the review keeps the cost ordering correct and the recorded verdict faithful to what the reviewer actually saw.
## [2026-06-19T21:48:31.765Z]
Add a fill-time prompt-size guard for companion-backed reviews. The deterministic prompt-size gate runs during read-only verification, but it reconstructs a companion's injected files from the previously recorded observations — which do not exist yet on a pair's very first fill, so an oversized companion prompt could reach the reviewer once before the gate engaged. The fill now measures the assembled prompt against the tier limit after resolving companions and before calling the reviewer, failing closed (nothing written, no reviewer call) when it overflows, so an over-limit companion pair behaves the same on first fill as on every later check.
