Positive closure now advances a node's source fingerprint and log baseline only when every enforced rule is freshly verified against the current code in this run and the node was not gated. ## [2026-06-13T07:02:06.114Z]
When a component lists a file that cannot be read, filling and closing verification state must fail closed rather than crash or quietly advance. Computing a component's source fingerprint over an unreadable file now raises a typed error; the fill and positive-closure steps catch it and decline to advance that component's recorded fingerprint or log baseline, leaving the component blocked rather than aborting the whole command with an unclassified error or closing over content that was never verified. The reasoning is that an unreadable listed file is a hard error for the author to fix, never grounds to record a passing state.
## [2026-06-13T07:27:16.966Z]
Agent-facing diagnostic messages (the what/why/next structure that tells an agent what went wrong and how to fix it) are formatted by the command-line layer, not by the engine. The fill stage previously built and wrote these messages itself; that formatting work has been moved out. The engine now emits each diagnostic as a structured object carrying the three fields and hands it to a sink supplied by the caller, which renders it; plain progress lines (counts, per-pair status) stay as direct writes because they are status, not remediation guidance. The reason for the split is that keeping all message formatting in one layer means presentation can change in a single place, and the engine stays a pure producer of structured results that any front end can render.

## [2026-06-14T22:16:34.150Z]
The fill stage no longer runs or persists the architecture relation-conformance check. That check became a live computation performed on every read of the graph, so running it again during the verdict-filling stage and writing its outcome into the committed lock would duplicate work and store a result that is always recomputed. The garbage-collection step that pruned stale relation results from the lock is likewise removed, because there is no longer any relation result stored to prune. Verdict filling now concerns itself only with the rule verdicts it actually persists.
## [2026-06-16T09:52:38.910Z]

LLM verdicts are now written to the lock the moment each pair completes, the same way deterministic checks already behaved, so interrupting a long approve keeps every finished verdict and the next run resumes only the rest; previously they were buffered and flushed only after the whole tier batch, so an interrupt discarded all completed reviewer work. ## [2026-06-20T11:18:08.370Z]
Added a deterministic-only fill mode. WHY: it is the keyless gate for CI and the pre-commit hook now that deterministic verdicts live in a gitignored local cache instead of the committed lock. In this mode the stage fills only the free deterministic pairs, skips LLM fills, and scopes every write to the gitignored cache so the committed files are never modified. Positive closure is skipped in this mode on purpose: recording a node's closure baseline writes a committed file, which a partial or CI run must never do. A fresh checkout has no cache, so this mode rematerializes it for free before a plain read can pass.
