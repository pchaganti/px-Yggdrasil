Introduced computeExpectedPairs and computeSourceFingerprint as the read-side foundation for the verdict lock. The engine needs one shared definition of a node's subject-file set so the future check and fill stages can compare stored verdicts against the expected pair universe without duplicating subject-file derivation logic across multiple files. The binary-file catalogue and the child carve-out helper (getChildMappingExclusions) were moved to this module because pairs.ts is the canonical site where subject files are assembled — keeping these concerns co-located prevents the three sites that previously each had their own notion of a node's effective files from diverging again. Verification units are now computed from effective aspects crossed with each aspect's declared review granularity (per: node vs per: file), with scope.files filtering applied before the unit split.

## [2026-06-12T13:53:48.024Z]
The scope.files filter in computeExpectedPairs was discarding files that evaluateFileWhen reported as unreadable (e.g. due to EACCES) without any signal to the caller. A file that cannot be read is excluded from the reviewer's subject set, which means if it was the only match, the aspect gets zero pairs — a vacuously green result despite a real gap in coverage. The system is specified fail-closed: every unreadable file under a content filter must surface as a blocking error. Changed computeExpectedPairs to return PairComputation {pairs, unreadable} instead of a bare array. Unreadable candidates are recorded in an UnreadableSubject[] with nodePath, aspectId, path, and the OS error reason. Callers are documented as required to surface non-empty unreadable as a blocking error. Deduplication prevents duplicate records for the same (nodePath, aspectId, path) triple.
