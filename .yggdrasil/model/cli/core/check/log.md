## [2026-05-29T17:08:10.071Z]
Cross-node files that a structure-shape rule reads through a related component now count toward the dependent component's verification identity and its change-impact blast radius. Previously such a rule could inspect a sibling component's file, but the set of files brought into the dependent component's drift fingerprint was only assembled when the stored approval baseline was supplied to the file-collection routine — and the routine that gates everyday drift checks did not supply it. The consequence was a silent gap: editing a file that only a neighbour's structure rule reads never marked the dependent component as needing re-verification, and the change-impact query reported zero affected components for that file. ## [2026-05-30T06:29:17.474Z]

The gate that reports drift must decide, for each changed file, whether the change is a direct source edit or an upstream cascade from a graph element. That decision previously relied on a lookup table built inline here, duplicated almost verbatim in the approve path. ## [2026-05-30T21:05:41.490Z]
Drift attribution was sharpened so a dependent node is no longer fully re-reviewed when it drifts only because a cross-node file read by one of its graph-aware deterministic checks changed. Previously such a change was un-attributable to any specific check, so the whole node — including its human-language reviewer aspects — was conservatively re-verified; that re-verification is paid and non-deterministic, so it produced flaky, costly churn on nodes whose own code never changed. Now the change is attributed to the specific deterministic check that reads the file (via the per-node record of which files each check touched), so only that local, free check re-runs and the human-reviewer verdicts carry forward unchanged. The motivation is correctness-preserving cost control: a check that only reads cross-node files cannot affect what a human-language reviewer sees, so re-running it alone is exactly right.

Specific to this node: the drift report that explains why a cascade fired now names the owning check for the synthetic per-check identity causes (the touched-file-set summary and the resolved-reviewer-tier summary), instead of attributing them to an unknown source.
