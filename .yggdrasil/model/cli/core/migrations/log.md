## [2026-05-28T14:30:01.418Z]
Added inspect-only addAspectStatusDefaults pass to to-5.0.0.ts that detects two v5-breaking patterns in pre-5 graphs and emits warnings withholding the version bump. First, escalation: an aspect with default enforced status implying another aspect whose own default is lower (advisory or draft) via a bare reference or object form without status_inherit. Under the new strictest propagation mode the implied aspect will silently run at the higher status when reached via the implier. Second, downgrade: an explicit per-attach-site status that is strictly below the cascading anchor (max of aspect-default and all other channels contributing the same aspect onto the same node). Both patterns are surfaced from raw YAML parsing without instantiating the full graph — keeps the migration robust to pre-v5 parse rules. Wired after the secrets pass so warnings accumulate alongside the existing reviewer/secret checks.

## [2026-05-29T18:22:43.091Z]
The v4-to-v5 config migration now validates the legacy global reviewer.consensus before copying it into per-tier blocks. In v4 the field had no constraint, so an even value (e.g. 2 or 4) was legal. In v5 each tier requires a positive odd integer for consensus. Silently copying an even value produced a structurally invalid v5 config that the parser rejected on load; because no warning was emitted the version was bumped and the migration could not re-run — forcing a manual hand-edit to recover. The fix mirrors the file's existing warning-prevents-migration convention: if reviewer.consensus is present and is not a positive odd integer, a descriptive warning is pushed and the function returns with changed: false and value: undefined, leaving both the config file and the version untouched so the user can correct the value and re-run the upgrade. An absent consensus or an odd value (including 1) continues to migrate normally as before.
