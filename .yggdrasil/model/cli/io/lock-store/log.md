## [2026-06-12T11:45:55.506Z]
Conflict-marker detection in readLock was changed from a substring check (raw.includes('<<<<<<<')) to a line-anchored regex (/^<<<<<<< /m.test(raw)). The motivation: verdict reason strings may legitimately contain the text '<<<<<<< HEAD' when a reviewer quotes merge-conflict fragments from the source being checked. A substring match would throw LockInvalidError on a valid, self-written lock file in that case. The line-anchored form matches only real git conflict markers, which always start a line and are immediately followed by a space — a JSON-escaped string value inside the file cannot satisfy this pattern. The behavioral guarantee is unchanged for actual conflict-markered files: a file with a real unresolved merge conflict still throws.
