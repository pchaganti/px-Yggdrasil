## [2026-06-29T05:24:52.926Z]
Introduced a single read-only gateway between the read-only audit portal and the engine. Previously the portal pipeline declared direct dependencies on a dozen separate engine subsystems — graph loading, the check runner, lock verification, pair computation, the relation analyzer, the suppression scanner, issue grouping, the effective-aspect cascade, reviewer-tier selection, and the log reader. That spider of edges meant any change to an internal subsystem could ripple into the portal, and there was no single place that guaranteed the portal stays read-only. This module is now the ONE place the portal reaches the engine: it wraps every needed call behind a clean read-only surface, loads the graph committed-only so secrets are structurally unreachable, and imports no lock writer. Concentrating the coupling here lets the read-only invariants (no secrets, no lock writer, committed-only load) be enforced mechanically on this one node, and lets the rest of the portal depend on just this facade plus the data contract.
## [2026-06-29T12:36:31.021Z]
Added three read-only provenance/freshness functions to the single engine seam: a content hash over the committed lock triad (nondeterministic verdicts + per-node logs baseline; the gitignored deterministic cache is excluded so the hash is stable across machines for one commit), a git HEAD commit-ref reader that follows .git/HEAD to its loose or packed ref and reports nothing rather than a fabricated ref outside a git tree, and a per-node source-freshness comparison against the committed fingerprint baseline. Freshness fires only where a committed baseline exists, because the engine records a source fingerprint only for log-required node types; a missing baseline is the normal case and must never be read as a change, so the portal cannot paint the whole repo unverified from absent baselines. The two cases a missing baseline leaves uncovered are already honest elsewhere — a node with reviewer pairs flips them to unverified on any input change, and an unguarded node is already its own distinct non-green state — so this signal adds exactly the one remaining case: a node whose green attests specific committed bytes that have since been edited.
