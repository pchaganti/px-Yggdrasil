## [2026-05-30T14:09:01.061Z]
The per-aspect language declaration was removed from the system. An aspect that ships a parsed-tree check used to carry a `language:` list naming the languages it targeted, and the system validated that list with four dedicated checks. Nothing in the runtime ever read it: the engine already determines each source file's language from its file extension through one shared registry, then loads the matching grammar. The declaration was therefore inert metadata that could silently disagree with what the engine actually parsed — an authoritative-looking field that governed nothing.

This change deletes the field, the validators that policed its shape, and the documentation and tests that described it, and promotes the extension-to-grammar registry to the single authority for matching a file to its parser. The motivation is to remove a confusing mismatch between what an aspect claimed about languages and what the engine did, and to collapse the duplicated extension-knowledge that had drifted into more than one place.

The language a parsed-tree check sees for a given file is now derived solely from that file's extension; an unrecognized extension yields no parsed tree rather than a per-aspect error. The drift identity of a graph-aware structural check is deliberately held stable across this change so existing approvals are not invalidated by metadata that never affected behavior.
## [2026-05-31T10:45:42.357Z]
Made the parser locate its grammar binaries correctly in a published install. The grammars are bundled with the package, but two layout assumptions were wrong: the binaries were copied under a short name the parser never looks up, and the directory the parser searched was computed one level too high for the flattened published layout — so a real install found no grammar and fell back to a build-only dependency that ships only to developers, failing with an opaque module-not-found. The parser now searches the directory beside the bundled entry first, keeps the previous location as a fallback, and degrades to a clear could-not-find-grammar message when no grammar package is present, instead of leaking a raw require error. This is what makes the local-language checks usable by an adopter rather than only in this repo's own test runs.

## [2026-06-08T19:56:24.041Z]
A deterministic check that only inspects file text must be able to see every file in a node's mapping, including files whose extension has no parser (documentation, shell scripts, JSON, and the like). The single-file deterministic runner previously parsed every file up front and aborted the entire run on the first file it could not parse, with a message that told the author to delete exactly the file the check was meant to read. That contradicted the promise made to check authors that content-only rules may iterate non-parseable files, and it disagreed with the graph-aware runner used elsewhere, which already hands such files to the check with no parse tree attached. The runner now treats a file with no registered grammar as non-parseable: it is still delivered to the check with its text and an absent parse tree, so text and pattern rules run over it, while a genuine failure to load a grammar that should exist still fails closed. Suppression markers, which are discovered through the parse tree, simply contribute nothing for a file that has none.
## [2026-06-08T20:29:14.366Z]
Inline suppression waivers must be honored in every source language, not only those with a parser. Markers were previously discovered solely by walking a file's parse tree for comment nodes, so a file whose extension has no registered grammar yielded no waivers at all. Once content-only deterministic checks began running over such files, an author flagged in, say, a SQL or shell file had no way to record an approved exception there — the escape hatch was missing for exactly the files that needed it. Suppression collection now falls back to a language-agnostic scan of the raw file text when there is no parse tree, keying on the distinctive suppression token rather than on any particular comment delimiter, so a waiver works regardless of language. Files that do have a parser keep using their comments, which keeps the token from being mistaken for a marker when it merely appears inside a string. The collector accordingly receives the file text alongside the optional tree.
## [2026-06-08T22:08:23.047Z]
Loading a parser grammar must be safe when many checks run at once. The one-time initialization of the parsing runtime and the loading of each language grammar were previously gated by a flag and a cache that were only set AFTER the awaited work completed. When several deterministic checks ran concurrently — as they do when approval verifies multiple targets in parallel — they could all observe the "not ready yet" state in the same window, all kick off their own initialization or grammar load, and one of them could pick up a grammar object that was still only half-loaded, which the parsing library reports as an incompatible language version. The symptom was a deterministic check spuriously failing in a batch while the very same check passed when the target was processed on its own. The fix records the in-flight work as a shared promise the moment it starts, so every concurrent caller awaits the same single initialization and the same single grammar load; a load that fails is removed so a later attempt can retry rather than inheriting a cached failure. Determinism is the whole point of these checks, so a concurrency-dependent failure had to be eliminated rather than worked around by serializing approvals.

## [2026-06-24T10:23:02.883Z]
Fix WASM heap exhaustion that caused Aborted() crash on large repos.

web-tree-sitter Parser and Tree objects are allocated in the WASM heap and require explicit .delete() to release memory — JavaScript GC does not free them. On repos with many files and aspects, accumulating hundreds of unreleased parsers and trees exhausted the heap, triggering an abort() call from within the WASM runtime that surfaced as an unclassified Aborted() error after all pairs were verified.

Two changes: (1) Parser objects are now cached per language rather than created fresh on every parseFile() call. Language objects were already cached; parser creation is synchronous once the Language is loaded, so one instance per language is safe to reuse across sequential parse() calls. (2) A destroyParseCache() helper was added to parse-cache.ts that iterates all entries and calls ast.delete(), clearing the map. Both ast/runner.ts (locally-parsed trees when no external cache is provided) and structure/runner.ts (locally-owned parse caches) now call this on exit via try/finally so cleanup runs on every code path including exceptions.
## [2026-06-25T15:24:58.777Z]
Fixed two WASM tree lifecycle bugs in the AST runtime layer.

In runner.ts, a Tree created by parseFile() was registered for local cleanup (localTrees.push) after a hasError guard that could throw AstRunnerError. If that guard fired, the tree was never pushed and therefore never deleted by the outer finally block, causing a permanent WASM heap leak. The fix moves localTrees.push to immediately after parseFile() succeeds, before any conditional branch that might exit early.

In parser.ts, added withParsedFile — a new exported function that parses a file, calls a callback with the resulting Tree, and guarantees tree.delete() in a finally block. This is the canonical WASM-safe API for bounded parse operations. All callers that do not need to transfer tree ownership should use withParsedFile instead of parseFile directly.

The root cause across all related bugs was that parseFile() returns a WASM-heap Tree whose lifecycle is invisible to the JavaScript garbage collector. Every call site had to manually remember to call .delete() inside a finally block — a discipline that cannot be enforced by the type system and was repeatedly forgotten. withParsedFile eliminates the manual discipline by encoding the lifecycle guarantee in the API itself.
